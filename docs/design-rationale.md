# 設計思想(Design Rationale)

本書は「なぜこの設計・このクレートにしたのか」「何を捨てたのか」「振り返って何を使うべきだったか」を記録する。構造の説明は [architecture.md](architecture.md)、実装上の落とし穴は [../learn/](../learn/cli-implementation.md) にあり、本書は **意思決定とトレードオフ** のみを扱う。

## 1. 何を最適化したか

設計全体を貫く優先順位は次の3つで、迷ったらこの順で判断した。

1. **反復コストをゼロに近づける** — エージェント開発は「プロンプト調整 → 実行 → 観察」の回数が品質を決める。LLM 既定をローカル Ollama にしたのも、全テストをオフラインで完結させたのも、GUI 開発中に core を再ビルドさせない構成にしたのも、すべて反復速度のため。
2. **LLM を信用しない** — 出力形式も、自己評価も、選んだ URL も信用しない前提で機械側の防御(寛容パース・閾値の二重判定・SSRF ガード・資源上限)を敷く。「賢いモデルなら不要な防御」はモデルを差し替えた瞬間に必要になる。
3. **フロントエンドを薄く保つ** — 価値はエンジンにあり、CLI/GUI は配線だけにする。これにより GUI 追加(後付け)が core への「イベント発行の追加」だけで済んだ。

## 2. アーキテクチャ上の判断

### 2.1 なぜ単一エージェント+自己評価ループか(マルチエージェントにしなかった理由)

[agentic-architecture.md](agentic-architecture.md) の調査結論そのままに、「強い単一ループを基本とし、必要になるまで分割しない」を採った。本タスクで具体的に効いた判断材料:

- 収集→評価→追加収集は **逐次依存** が強く、並列サブエージェント化してもコンテキスト分離の利益が薄い(評価者は常に全 findings を見る必要がある)。
- ローカル 3B モデルでは「複数エージェントの調整プロトコル」自体が最大の故障点になる(MAST の失敗分類のうち仕様・調整起因が大半、という知見と一致)。
- 評価役は別「エージェント」ではなく **同一 LLM の別プロンプト** で十分。生成と批評で文脈を分けることが本質であり、プロセスやモデルを分けることは本質ではない。

### 2.2 なぜ trait 境界はちょうど3つか

`LlmClient` / `SearchProvider` / `PageFetcher` だけを抽象化し、planner / evaluator / reporter は具象関数のままにした。

- 3つは **I/O 境界**(プロセス外と話す場所)であり、テストで偽物に差し替える必要が必然的にある。
- 一方 planner/evaluator は「同じ LLM に違うプロンプトを渡す」だけで、変化軸はプロンプト文字列にある。trait にすると抽象の層だけ増えて変更点(prompts.rs)が分散する。**「変わりうる軸」と「抽象化する場所」を一致させる** のが方針。やりすぎの抽象化は、この規模では trait 定義の管理コストが利益を上回る。

### 2.3 評価を3軸+二重判定にした理由

要件(新しいか・間違っていないか・網羅的か)を鮮度/正確性/網羅性のスコアにそのまま写像した。設計上のポイントは軸の数ではなく **終了判定を LLM に独占させない** こと:

- `is_sufficient`(LLM の総合判断)**かつ** 全軸スコア閾値以上、の AND を取る。LLM-as-a-Judge は「総合的には十分」と言いながら個別軸に低スコアを付ける非整合を平気で起こすため、機械側で矛盾を拒否する。
- さらに「進捗ゼロ(新規 finding なし・追加クエリなし)での早期終了」と「反復回数上限」を独立に置く。**停止保証を LLM の協力に依存させない** — これは課金 API に切り替えた際のコスト暴走防止でもある。

### 2.4 ワークスペース分割を「GUI 追加時」に行った理由

最初から3クレートにせず、単一クレートで完成させてから分割した。先行分割は境界を推測で引くことになるが、後からの分割は「CLI が実際に使っている公開面」をそのまま lib の API にできる。gpui(依存約400クレート)を別クレートに隔離したことで、core/cli の開発ループに GUI のビルドコストが一切乗らない。

### 2.5 進捗イベントを `Box<dyn Fn>` にした理由

core から GUI への進捗通知は、チャネル型(tokio mpsc 等)を core の API に出す案もあったが、コールバック(`EventSink = Box<dyn Fn(AgentEvent) + Send + Sync>`)にした。チャネルを公開すると core が **特定ランタイムの型に縛られる**。コールバックなら CLI は無視、GUI は「チャネルへ転送する closure」を渡すだけで、core は gpui にも tokio のチャネル事情にも無知でいられる。イベントは「ログの構造化版」であり、失敗してもエージェントを止めない(送信エラーは握りつぶす)非機能経路として扱う。

### 2.6 日本語レポートを「直接生成」にした理由(後段翻訳との比較)

「英語で推論させて最後に日本語へ翻訳する方が品質が出るのでは」という論点に対し、**収集・評価は元言語のまま、レポート合成の1回だけ日本語で出力**という折衷を採った。

- 独立した翻訳パスを設けると LLM 呼び出しが1回増え(時間・コスト)、翻訳時に引用番号 `[n]` や URL が破損するリスクを新たに抱える。
- 「推論は英語が得意」という性質は、中間生成物(findings・評価 JSON)を元言語のまま保つことですでに享受している。レポーターは findings を読んで日本語で書き起こすため、実質「推論済みの内容を日本語で表現する」だけになり、翻訳専用パスとの品質差は小さい。
- 大型モデル(Claude / OpenAI)では直接生成で品質問題はなく、小型モデルで日本語が崩れる場合の実験用に `AGS_REPORT_LANGUAGE` で言語を切替可能にしてある(2段変換を試すならここを `English` にして外部翻訳を挟む構成も取れる)。

### 2.7 監査トレースを「イベントの JSONL 併置」にした理由

運用・監査要件(途中評価や追加検索の事後追跡)は、(a) tracing ログの保存、(b) DB への実行記録、(c) 進捗イベントの永続化、の3案を比較し (c) にした。GUI 進捗表示用の `AgentEvent` がすでに「計画・クエリ実行・ページ取得・評価(指摘事項・追加クエリ込み)」を網羅しており、**Serialize を足すだけで監査ログになる** — 表示と監査で情報源が一本化され、二重実装も将来の乖離もない。形式を JSON Lines にしたのは、追記的な時系列データの標準形で `jq` 等でそのまま処理でき、1行壊れても残りが読める(寛容パース `from_jsonl`)ため。レポート・メタデータと同じ stem で併置することで「削除」も自然に3ファイル一括になる。

### 2.8 レポート履歴をファイル(md + json サイドカー)にした理由

SQLite が常套だが、件数は高々数百・クエリは「新しい順に全件」だけ・ユーザーが Markdown を直接 Finder や他ツールで開けることに価値がある、の3点でプレーンファイルにした。本文(.md)とメタデータ(.json)を分けたのは、本文をエディタで開いても履歴一覧が壊れないようにするため。全文検索やタグが要件になった時点が SQLite への移行点。

## 3. クレート選定の理由と代替案

| 採用 | 代替案 | 判断 |
|---|---|---|
| tokio | async-std, smol | reqwest/エコシステムの事実上の前提。選択の余地は実質ない |
| reqwest + rustls | hyper 直, ureq, curl | ureq(同期)はランタイム不要で魅力だが、ページ取得の並列化余地と tokio 前提の周辺都合で reqwest。rustls は OpenSSL というシステム依存・ビルド差異を消すため |
| thiserror + anyhow | anyhow 単独, snafu | 「ライブラリは型付き・バイナリは文脈付き」の標準的分業。フロントエンドが `BlockedUrl` を識別できることに意味がある |
| scraper | html5ever 直, kuchiki, tl | CSS セレクタで十分な抽出要件。html5ever 直は低レベルすぎ、kuchiki はメンテ停滞 |
| clap (derive) | argh, pico-args | ヘルプ生成・エラー品質で標準。バイナリサイズ最適化が要件なら pico-args |
| sha2 | ahash/fxhash, md5 | finding の同一性判定は「安定した内容アドレス」が目的で速度は無関係。非暗号ハッシュにする理由がない |
| chrono | time, jiff | serde 連携と書式の慣れ。新規なら jiff も有力だった |
| tracing | log + env_logger | 構造化フィールド(`query = %q` 等)がエージェントの観測性に直結。GUI のイベント設計も tracing のフィールド感覚をそのまま流用した |
| DuckDuckGo HTML | Brave/Tavily/SerpAPI, Google CSE | **キー不要で初回体験がゼロ設定** を最優先。スクレイピングの脆さは SearchProvider trait + SearXNG 代替で保険。商用運用なら Tavily 等の検索 API に差し替えるべき(trait 1実装の追加で済む) |
| Ollama 既定 | llama.cpp 直, LM Studio, candle | プロセス分離された HTTP API・モデル管理込み・OpenAI 互換でない素の API も素直、で運用が最も軽い。candle(Rust 内推論)は配布は綺麗になるがモデル管理を自前で背負う |

### GUI: gpui + gpui-component を選んだ理由(指定要件+α)

gpui はユーザー要件だが、採用にあたり比較した代替と評価:

| 候補 | 評価 |
|---|---|
| **gpui**(採用) | GPU 描画でネイティブ品質、Rust 単一言語で core と直結。弱点はドキュメント不足と API 流動性(learn/gui-implementation.md 参照) |
| Tauri | Web 技術で UI を書けて最も生産的だが、フロントが TS になり「Rust で完結」が崩れる。WebView 分のフットプリント |
| egui | 即時モード GUI。プロトタイプは最速だが、日本語 IME・テキスト選択・リッチテキストが弱く「レポート閲覧」要件に合わない |
| iced | Elm アーキテクチャで堅実だが、Markdown 表示・高機能入力を自前で組む量が gpui-component 利用より多い |
| Slint | 宣言 UI 言語(.slint)の学習が別途必要。商用ライセンス考慮も発生 |

gpui 単体ではなく **gpui-component を併用** したのが実質的な決定打。素の gpui にはテキスト入力すら無く(Zed は自前エディタを持つため)、日本語 IME 対応入力と Markdown レンダラを自作すると GUI 本体より大きくなる。「フレームワークの素体」と「ウィジェット層」を別プロジェクトから組み合わせるのが 2026 年時点の gpui エコシステムの現実的な使い方。

## 4. 採用しなかった/次に使うべき選択肢(振り返り)

実装を終えた時点での正直な反省点。優先度順。

1. **本文抽出に readability 系を使うべきだった** — 現在の「全タグからテキスト収集」はナビゲーションやフッターのノイズを LLM に渡しており、小型モデルの抽出品質を確実に下げている。`readability` / `dom_smoothie` 系(Firefox Reader View 相当)に置き換えれば、`html_to_text` の差し替えだけで findings の質が上がるはず。最初からこちらでもコストは大差なかった。
2. **構造化出力は JSON Schema 制約を使うべき段階に来ている** — Ollama は `format` にスキーマそのものを渡せる(OpenAI も structured outputs 対応)。現在の「寛容パース+サルベージ」は防御として残す価値があるが、入口でスキーマ制約をかければサルベージの発動頻度自体を減らせた。
3. **HTTP リトライがない** — 一時的な 5xx やタイムアウトを即「ページスキップ」にしている。`backon` 等で指数バックオフ1〜2回を入れるだけで取得成功率は目に見えて上がるはず。ただしエージェントは「別ソースを探す」自律性で部分的に自己回復するため、致命傷にはなっていない。
4. **重複排除が字面ベース** — 正規化ハッシュは「ほぼ同文」しか弾けず、言い換えられた同内容は通す。`fastembed` などローカル埋め込みでの類似度判定が次の一手。ただし 3B モデル環境に埋め込みモデルを足す運用コストと見合うかは要検討で、初版で見送ったこと自体は妥当だった。
5. **GUI のレポート生成がストリーミングでない** — レポート合成中は進捗が止まって見える。LLM クライアントに streaming を足し、`AgentEvent::ReportChunk` を流せば体感が大きく変わる。trait に `complete_stream` を足す拡張で実現できる設計にはなっている。
6. **採用しなくて正解だったもの**: LangChain 的フレームワークの Rust 移植(rig 等)— 本体ループは200行弱であり、フレームワークの抽象を学ぶコストの方が高い。エージェントループは「自分で書いた方が薄い」規模だった。
