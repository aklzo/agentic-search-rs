# セキュリティ

エージェントは「LLM が選んだ URL に自動でアクセスする」性質上、通常の HTTP クライアントより厳しい防御が必要になる。実装済みの対策と、変更時に守るべき不変条件をまとめる。

## SSRF 対策(`crates/core/src/fetch/guard.rs`)

検索結果の URL は信頼できない入力として扱い、取得前に二段階で検証する。

1. **静的検証 `validate_url`**(同期)
   - スキームは `http` / `https` のみ許可(`file:` 等を遮断)
   - URL 埋め込み認証情報(`user:pass@`)を拒否
   - IP リテラルはループバック・プライベート・リンクローカル・CGNAT・未指定などの非公開レンジを拒否(IPv4-mapped IPv6 を含む)
   - `localhost` / `*.local` / `*.internal` / ドットなしホスト名を拒否
2. **DNS 検証 `ensure_public_host`**(非同期)
   - ホスト名を解決し、**解決先のすべてのアドレス**が公開 IP であることを確認(内部 IP を指す DNS レコード対策)
3. **リダイレクト再検証**(`crates/core/src/fetch/mod.rs` の custom redirect policy)
   - 各リダイレクトホップの URL を再度 `validate_url` に通す(公開 URL → `Location: http://169.254.169.254/...` のようなバウンス対策)
   - リダイレクトは最大5回

**不変条件**: 新しい取得経路(別のフェッチャ実装など)を追加するときも、必ずこのガードを通すこと。

## リソース上限

外部入力によるリソース枯渇を防ぐため、すべての取得・生成に上限がある(値は [configuration.md](configuration.md) 参照)。

- レスポンス本文はストリーム読みで 2 MiB に打ち切り(`Content-Length` 詐称に依存しない)
- テキスト系 Content-Type 以外(PDF・画像など)は取得しない
- 接続・全体タイムアウト、反復回数・クエリ数・ページ数の上限
- TLS は rustls(OpenSSL 非依存)

## 秘密情報の扱い

- API キーは環境変数からのみ読む。CLI 引数では受け取らない(プロセス一覧に露出するため)
- `SecretKey` 型(`crates/core/src/config.rs`)が `Debug` 出力を `***` にマスクし、設定をログに出してもキーが漏れない
- ログ(tracing)にページ本文や LLM 応答の全文は出さない

## プロンプトインジェクションについて

取得した Web ページの本文は LLM のプロンプトに入るため、**ページ側の指示文がエージェントの挙動を汚染しうる**。現在の緩和策:

- エージェントのアクション空間が狭い(検索クエリの発行とページ取得のみ。シェル実行・ファイル書き込み等のツールを持たない)
- 取得先はガード済みの公開 Web のみで、注入が起きても到達できる範囲が限定される
- 抽出プロンプトは「ページに書かれていない事実を作らない」「関連がなければ空リストを返す」よう指示している

レポートの記述はあくまで「収集した Web 情報の要約」であり、検証済みの事実ではない。レポート末尾の自己評価(正確性スコアと既知の問題)を必ず併記する設計はこのため。

## 変更時のチェックリスト

- [ ] 新しい外部アクセスはガード(SSRF・サイズ・タイムアウト)を通っているか
- [ ] 新しい設定値に秘密情報が含まれるなら `SecretKey` を使っているか
- [ ] エラーメッセージ・ログに URL 以外の生データ(本文・キー)を含めていないか
- [ ] `cargo clippy --all-targets -- -D warnings` が通るか
