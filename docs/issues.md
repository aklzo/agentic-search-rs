## 詳細

<a id="i-01"></a>
### I-01 検索・取得処理が完全逐次(並列化未実装)

**背景**: `run_iteration` はクエリを、`gatherer::gather` はページを `for` ループで1件ずつ `await` しており、tokio は「待ちでブロックしない」効率しか活かせていない。実測でページ取得(2〜4秒/枚)だけで約30〜40秒/反復を消費。実サービス(数十ページ同時取得)と比べ検索側は遅い。

**選択肢**:
1. ページ取得の並列化(`buffer_unordered`、同時4〜8本)— 取得先は多様なホストで安全。効果大
2. 取得と LLM 抽出のパイプライン化 — ローカル LLM で特に有効
3. 抽出(LLM 呼び出し)の並列化 — API プロバイダのみ有効。ローカルは GPU 飽和で逆効果
4. 検索クエリ自体の並列化 — DuckDuckGo の 429/CAPTCHA リスクがあり**非推奨**

**推奨**: 1 → 2 の順。ただし 3 を含め **API レートリミットとの兼ね合いの設計が先**。

**経緯**: 2026-06-12、レートリミット懸念により**現時点では改修不要と判断(保留)**。再開条件: 実行時間が運用上の問題になったとき。同時実行数の上限を `Limits` に持たせる設計とセットで行うこと。

<a id="i-02"></a>
### I-02 ローカル SLM の prefill 律速

**背景**: 本ツールの LLM 呼び出しは「数千トークン入力 → 数百トークン出力」の入力過多型で、ローカル推論の弱点である prefill(計算律速)を直撃する。Sonnet 4.6(API)が処理量3倍でもローカル 3B の半分の時間で完走した主因(反復回数は同一で、モデルの賢さによる時短ではない)。

**選択肢**:
1. `OLLAMA_FLASH_ATTENTION=1` / KV キャッシュ量子化(設定のみ・小〜中の効果)
2. プロンプトの安定部分先頭化によるプレフィックスキャッシュ活用 — 実装済みの構造だが、支配項のページ本文は毎回異なるため効果は限定的
3. I-03(readability)による入力削減 — 品質を落とさない削減で実質的な本命
4. MLX 等 NPU 系ランタイムへの切替 — 効果は大きいが LlmClient 実装の追加が必要

**推奨**: 構造的制約として受け入れ、3(I-03)と 1 で緩和。デバイス組み込みの文脈では専用ランタイム+NPU が業界標準の解。

<a id="i-03"></a>
### I-03 本文抽出のノイズ(readability 系未導入)

**背景**: 現在の `html_to_text` は script/style 等を除くだけで、ナビゲーション・フッター等のノイズを LLM に渡している。小型モデルの抽出品質を確実に下げており、I-02 の入力量も無駄に増やしている。design-rationale の振り返りでも最優先候補。

**選択肢**: `readability` / `dom_smoothie` 系クレート(Firefox Reader View 相当)へ `html_to_text` を差し替え。失敗時(本文検出不能)は現行ロジックへフォールバック。

**推奨**: 実施。差し替え箇所は `fetch/extract.rs` に閉じており、フィクスチャテストで品質比較も容易。

<a id="i-04"></a>
### I-04 学術ソース向け検索の欠如

**背景**: DuckDuckGo は GitHub・技術ブログには届くが、論文の網羅(「2025-2026 の最新研究を調査」のような要求)が弱い。Style-Bert-VITS2 実験でも arXiv 直接のヒットはほぼなかった。

**選択肢**: arXiv API / Semantic Scholar API を `SearchProvider` として追加し、プランナーまたは設定でソースを選択。trait 設計により1実装の追加で済む。

**推奨**: 研究調査ユースケースを重視するなら次の機能追加候補。クエリの性質からソースを選ぶ判断をプランナーに委ねる設計(クエリに `source: web|paper` を付けさせる)が拡張として自然。

<a id="i-05"></a>
### I-05 HTTP リトライがない

**背景**: 一時的な 5xx・タイムアウトを即「ページスキップ」にしている。エージェントが別ソースを探す自律性で部分回復するため致命傷ではないが、取得成功率を下げている。

**選択肢**: `backon` 等で指数バックオフ1〜2回(取得・検索・LLM の各クライアント)。

**推奨**: 実施(小規模)。リトライは冪等な GET と LLM 呼び出しに限定し、回数・間隔は `Limits` に置く。

<a id="i-06"></a>
### I-06 JSON Schema 制約付き出力を未使用

**背景**: Ollama は `format` にスキーマ自体を渡せ、OpenAI は structured outputs に対応。現在はプロンプト指示+寛容パース(`llm/json.rs`・項目単位サルベージ)で守っているが、入口で制約すればサルベージの発動頻度自体を減らせる。

**選択肢**: `ChatRequest` の `json_mode: bool` を `json_schema: Option<Value>` に拡張し、対応プロバイダはネイティブ制約・非対応(Claude)はプロンプト注入。寛容パースは防御として残す。

**推奨**: 小型ローカルモデルを主力にするなら効果大。Claude 主体なら優先度は下がる。

<a id="i-07"></a>
### I-07 重複排除が字面ベース

**背景**: 正規化ハッシュは「ほぼ同文」しか弾けず、言い換えられた同内容の findings を通す。レポートの冗長さの一因。

**選択肢**: `fastembed` 等のローカル埋め込みで類似度判定。ただし 3B 運用環境に埋め込みモデルを足す運用コストと要相談。

**推奨**: 保留寄りの未着手。レポートの冗長性が実害になってから。

<a id="i-08"></a>
### I-08 レポート生成がストリーミングでない

**背景**: レポート合成中(数十秒)は GUI の進捗表示が止まって見える。

**選択肢**: `LlmClient` に `complete_stream` を追加し、`AgentEvent::ReportChunk` を流す。trait 拡張で実現できる設計にはなっている。

**推奨**: UX 改善として価値はあるが機能追加が一段落してから。

<a id="i-09"></a>
### I-09 評価者の採点基準がモデル間で異なる

**背景**: 同一プロンプト・同一収集条件でも、3B は網羅性80(甘い)・Sonnet 4.6 は68(辛い)と採点基準が異なる。Sonnet では事実上反復上限まで回り続けるため、反復数=コストの予測がモデル依存になる。

**選択肢**: 閾値(現在70)のモデル別調整 / 評価プロンプトに採点基準のアンカー(各スコア帯の具体例)を追加 / 「スコア改善が頭打ちなら十分とみなす」収束判定の追加。

**推奨**: まずは観察継続。複数モデルでの実行データが溜まったらアンカー方式を試す。

<a id="i-10"></a>
### I-10 エージェント機構の高度化(3点セット)

**背景**: 3B 実験後に検討した3案。(a) re-planning(計画自体の書き直し)、(b) findings キュレーション(誤り判明分の削除)、(c) サブ質問の下流活用(軸ごとのカバレッジ追跡)。

**経緯と判断**: 2026-06-12 の Sonnet 4.6 検証で、観測されていた失敗(クエリ崩壊・仮説クエリ欠如・ランキング不在)は**すべてモデル能力起因**と確定。followup クエリ経由の軌道修正が十分機能したため、**3点とも保留**。

- 再開条件: 実行トレース(`.trace.jsonl`)に「追加クエリでは直せなかった失敗」が実際に記録されたとき
- 着手する場合の優先順: (c) → (a) → (b)。(b) は ACE 論文の指摘する context collapse(正しい事実の誤削除)リスクがあり最後
- 関連: [design-rationale.md](design-rationale.md) §2(なぜ単純なループを保つか)

<a id="i-11"></a>
### I-11 DuckDuckGo スクレイピング依存

**背景**: キーレスで初回体験が良い一方、(1) HTML 構造変更で壊れる、(2) アクセス頻度を上げると 429/CAPTCHA、(3) 商用利用のグレーさ、という脆弱性を抱える。I-01 の並列化を見送った直接の理由でもある。

**選択肢**: Tavily / Brave Search 等の商用 Search API を `SearchProvider` として追加(キー必須だが安定・高速・並列可)。SearXNG セルフホストは既に代替として実装済み。

**推奨**: 本格運用(並列化や高頻度実行)に進む際は商用 API 追加が前提条件。

<a id="i-12"></a>
### I-12 CLI に実行トレース保存がない

**背景**: 監査トレース(JSONL)は GUI のみ保存。トレースの読み書き(`events::to_jsonl` / `from_jsonl`)は core にあるため、CLI 側は薄い配線で済む。

**選択肢**: `--trace <path>` フラグを追加し、EventSink から JSONL を書き出す。

**推奨**: 運用で CLI 実行を監査対象にする必要が出たら実施(~30行)。

<a id="i-13"></a>
### I-13 拡散 LM(DiffusionGemma)の導入検討

**背景**: Google が 2026-06-10 に公開した DiffusionGemma(26B MoE・アクティブ 3.8B・Apache 2.0)は、テキスト拡散により生成(decode)を最大4倍高速化すると謳う。ローカル実行の高速化候補として検討した。

**調査結果(2026-06-12 時点)**:
- **Ollama では使用不可**。GGUF は公開済み(unsloth)だが、`diffusion-gemma` アーキテクチャを Ollama がロードできない([ollama#16664](https://github.com/ollama/ollama/issues/16664))。llama.cpp 側も対応 PR([#24427](https://github.com/ggml-org/llama.cpp/pull/24427))が未マージで、マージ後も専用 `llama-diffusion-cli` のみ対応(`llama-server` の HTTP API 非対応)のため、本ツールのクライアントから叩けない
- Apple Silicon では mlx-vlm / oMLX(サードパーティ製サーバー)に初期対応があるが「キャッシュなし」の実験的実装

**期待効果の分析(導入できても本ツールでは速くならない見込み)**:
- 拡散が速くするのは **decode** であり、本ツールのワークロードは **prefill 律速**(I-02)。decode は呼び出し時間の2〜3割しかなく、そこが仮に4倍速でも全体短縮は1割強
- 公称 4x は NVIDIA 専用 GPU(H100/RTX5090)での数値。拡散は「ボトルネックを帯域から**計算**へ移す」技術であり、Apple Silicon の弱点はまさに計算(prefill が計算律速)— M3 では効果が出ない方向
- メモリ 4bit で ~15-18GB と、llama3.2:3b(2GB)の7倍超。M3 24GB では他プロセスと競合しスワップリスク
- 精度は 26B MoE なので 3B より上の見込み(ただし Google 自身が「品質は自己回帰の Gemma 4 が上」と明言)— 精度目的でも速度目的でも中途半端

**推奨**: 見送り。decode 律速のワークロード(チャット・長文生成)向けの技術であり、本ツールの入力過多型には合わない。

**再開条件**: llama.cpp PR #24427 マージ+Ollama 対応(`ollama pull` で動く状態)になり、かつレポート合成(出力が長い唯一の工程)の高速化が課題になったとき。
